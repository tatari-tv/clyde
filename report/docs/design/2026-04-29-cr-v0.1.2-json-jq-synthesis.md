# Design Document: cr v0.1.2 â€” JSON report, jq required, prompt synthesis rules

**Author:** Scott Idler
**Date:** 2026-04-29
**Status:** Implemented (Architect-reviewed)
**Review Passes Completed:** 5/5 + Architect review (1 round, 4 findings, 3 actioned, 1 conceded with rationale)

## Summary

cr v0.1.2 makes three coordinated changes: (1) the report format switches from YAML to JSON so it is queryable with `jq`, the more common tool than `yq`; (2) `jq` becomes a required external tool, listed in `--help` and enforced at startup of `cr collect` with a helpful install hint; (3) the Opus rendering prompt gains a synthesis rule that forbids per-session dollar enumeration so the markdown writeup talks about themes instead of tagging every session title with its cost. A small `--help` formatting fix (separate the "default subcommand" line from the "logs path" line) rides along.

## Problem Statement

### Background

cr v0.1.1 (the version that just shipped) emits its report as YAML and renders to markdown via Opus using a prompt at `templates/report.pmt`. The `cr render` markdown for April 2026 (429 sessions, $4,437.78 total) revealed three pain points:

1. **Querying the report.** YAML is queryable with `yq`, but `yq` is less universally installed than `jq`. The user explicitly asked to switch to JSON so `jq` works directly.
2. **Markdown prose.** The Opus-generated markdown ended up as a bulleted list of session titles each tagged with its dollar amount, e.g. `read design document and survey codebase ($72.23), analyze codebase structure before implementation ($57.18), ...`. The current prompt's "Hard prohibition" only bans speculative quantification (cost-vs-engineer, hours-saved); it does not ban per-session dollar enumeration. The model takes the path of least resistance and enumerates.
3. **`--help` formatting.** The "REQUIRED TOOLS" footer ends with one line that mashes two pieces together: `Default subcommand is `collect`. Logs: ~/.local/share/claude-report/logs/claude-report.log`. Two distinct facts, one line.

`yq` is installed on the author's machine, so v0.1.1 works for the author today. But the report's value as a reusable tool depends on consumers being able to query it without installing a less-common dependency, and on the markdown writeup being readable rather than enumerative.

### Problem

- The report format ties consumers to `yq`, which is less common than `jq`.
- The markdown writeup is dollar-amount-per-line and reads as a ledger, not as a synthesis.
- A small `--help` line is two facts compressed into one.

### Goals

- Switch the report serialization from YAML to JSON, end to end (collect, render, prompt context block, tests, default output filename).
- Make `jq` a required external tool: list it in `--help`, fail `cr collect` at startup with an install hint if it is missing.
- Replace the prompt clause that allows per-session dollar enumeration with a synthesis rule that caps representative numbers per section and forbids the ledger pattern.
- Split the "default subcommand" and "logs" lines in `--help`.
- Drop the `serde_yaml` dependency entirely; nothing in cr will use YAML after this change.

### Non-Goals

- No new subcommands, no new flags beyond what is required for the above.
- No support for reading old YAML reports. Clean break: v0.1.2 only reads its own JSON output. Users regenerate by running `cr collect` again.
- No JSON schema versioning beyond the existing `schema-version: 1` field. Consumers who care will gate on that field.
- No change to the data model itself. The `Report`, `Totals`, `SessionEntry`, `ModelTokens` structs remain the same; only the serializer changes.
- No `cr render` failure when `jq` is missing. `jq` is a producer-side query tool; `cr render` reads JSON via `serde_json` directly.

## Proposed Solution

### Overview

Three commits on `main`, then `bump -a` to v0.1.2.

1. `refactor: emit report as JSON instead of YAML` (the format switch + dependency drop)
2. `feat(cli): require jq, separate default-subcommand from logs in --help` (the tooling change + help split)
3. `docs(prompt): force synthesis over per-session dollar enumeration` (the prompt rewrite)

Each commit is independently buildable and `otto ci`-green. The bump produces v0.1.2 and tags it on `main` (per `git.md`: only main, annotated tags only).

### Architecture

No structural changes. The data flow is unchanged:

```
cr collect:
  ~/.claude/projects/**/*.jsonl
    -> parse (claude-pricing)
    -> session::fold
    -> report::write_json     (was write_yaml)
    -> ./claude-report.json   (was .yml; same DEFAULT_OUTPUT constant)

cr render:
  ./claude-report.json        (default input; -i overrides)
    -> render::run reads JSON via serde_json
    -> build_context_block emits JSON
    -> summarize::opus wraps in ```json``` fence
    -> Opus + report.pmt
    -> ./<YYYY-MM>-claude-report.md   (render output; period-prefixed; unchanged)
```

The default collect output and the default render input share one `DEFAULT_OUTPUT` constant in `src/config.rs:47`; switching that constant from `./claude-report.yml` to `./claude-report.json` covers both directions in one edit.

The library boundary at `claude-pricing` is unaffected; only the report shape inside cr changes.

### Data Model

The `Report` struct is unchanged. Serialization changes from `serde_yaml::to_string` to `serde_json::to_string_pretty`. `#[serde(rename_all = "kebab-case")]` is preserved, so JSON keys are kebab-case (`"schema-version": 1`, `"spend-usd": 4437.78`, etc.).

Example JSON shape:

```json
{
  "schema-version": 1,
  "generated": "2026-04-29T03:14:00Z",
  "host": "desk",
  "since": "2026-04-01T07:00:00Z",
  "until": "2026-04-29T07:00:00Z",
  "totals": {
    "sessions": 429,
    "spend-usd": 4437.78,
    "untracked-models": ["<synthetic>"],
    "models": {
      "claude-opus-4-7": {
        "input": 12345,
        "output": 67890,
        "cache-5m-write": 0,
        "cache-1h-write": 0,
        "cache-read": 0,
        "total": 80235,
        "spend-usd": 2108.63
      }
    }
  },
  "sessions": {
    "<sid>": {
      "title": "...",
      "repo": "...",
      "begin": "...",
      "end": "...",
      "spend-usd": 12.34,
      "untracked-models": [],
      "jsonl-paths": ["..."],
      "models": { "...": { "..." : "..." } }
    }
  }

```

`jq` queries that the user will routinely run:

```sh
# Top 10 sessions by spend
jq '.sessions | to_entries | sort_by(.value."spend-usd") | reverse | .[:10]
    | map({sid: .key, spend: .value."spend-usd", title: .value.title})' \
    2026-04-claude-report.json

# Total spend per model
jq '.totals.models | to_entries
    | map([.key, .value."spend-usd"])' \
    2026-04-claude-report.json

# Sessions per repo, sorted desc
jq '.sessions | to_entries | group_by(.value.repo)
    | map({repo: .[0].value.repo, count: length})
    | sort_by(.count) | reverse' \
    2026-04-claude-report.json
```

### API Design

**`cr` CLI surface:** unchanged. No new flags. The defaults shift one extension:

```
cr collect              # writes ./<YYYY-MM>-claude-report.json (was .yml)
cr render               # reads ./<YYYY-MM>-claude-report.json (was .yml)
cr render -i path.json  # reads any JSON path
```

**`--help` REQUIRED TOOLS list adds jq:**

```
REQUIRED TOOLS:
  âœ… persona    1.0.14  (cr render: persona block in context)
  âœ… pandoc   3.1.11.1  (cr render --pdf)
  âœ… git        2.51.0  (cr collect: repo detection)
  âœ… jq           1.7.1  (cr collect: query JSON report output)

Default subcommand is `collect`.
Logs: ~/.local/share/claude-report/logs/claude-report.log
```

**Startup check** lives in `src/main.rs`, gated to the Collect subcommand. Putting it in `main.rs` rather than `lib.rs::run_collect` is deliberate: tests call `crate::run(&cfg)` (the lib entry point) and we don't want them to fail when `jq` is missing on a contributor's machine. The check belongs at the binary boundary, not the library boundary.

```rust
// src/main.rs, after Config::try_from(cli)
if let ResolvedCommand::Collect(_) = config.command
    && which::which("jq").is_err()
{
    eprintln!(
        "cr collect: jq is required to query the JSON report output but was not found on PATH.\n\
         Install: brew install jq  (macOS) | apt install jq  (Debian/Ubuntu) | dnf install jq  (Fedora)"
    );
    return Ok(ExitCode::from(2));
}
```

`cr render` does not check for `jq`; `serde_json` reads the file directly.

### Implementation Plan

#### Phase 1: Switch report serialization to JSON
**Model:** sonnet

Mechanical translation from `serde_yaml` to `serde_json` end to end.

- `src/report.rs`: rename `write_yaml` -> `write_json`, switch to `serde_json::to_string_pretty`. `load_existing_titles` reads JSON.
- `src/lib.rs`: caller updated.
- `src/config.rs`: `DEFAULT_OUTPUT` from `./claude-report.yml` to `./claude-report.json`. Doc comment "input YAML" -> "input JSON".
- `src/render.rs`:
  - `serde_yaml::from_str` -> `serde_json::from_str` for reading the report.
  - **Legacy YAML extension guard**: before parsing, if `cfg.input` ends in `.yml` or `.yaml`, bail with a guided error: `"input file ends in .yml/.yaml; cr v0.1.2+ emits and reads JSON. Re-run cr collect to regenerate as .json."` This catches the common mistake of `cr render -i ./claude-report.yml` from a stale alias or script and gives a clear remediation instead of a generic `serde_json::from_str` parse error.
  - `build_context_block` emits a JSON object via a small wrapper struct:

    ```rust
    #[derive(Serialize)]
    #[serde(rename_all = "kebab-case")]
    struct ContextBlock<'a> {
        persona: &'a PersonaBlock,
        options: ContextOptions,
        report: &'a Report,
    }
    #[derive(Serialize)]
    #[serde(rename_all = "kebab-case")]
    struct ContextOptions { include_tradeoffs: bool }
    ```

    Use **`serde_json::to_string`** (compact, single-line) for the context block sent to Opus, NOT `to_string_pretty`. Compact JSON keeps the LLM token cost as low as possible; the on-disk report file uses `to_string_pretty` separately for human/jq readability. The function signature changes from `(report_yaml: &str, ..., persona: Option<&PersonaBlock>) -> String` to `(report: &Report, ..., persona: Option<&PersonaBlock>) -> Result<String>`. Caller in `render::run` already has the parsed `Report` and passes it directly. Empty persona is `&PersonaBlock::default()` which serializes to `{}` due to `skip_serializing_if = Option::is_none` on every field.
- `src/summarize.rs`: fence change ` ```yaml ` -> ` ```json `. Variable rename. Internal log message rename.
- `src/cli.rs`: `about` string "YAML report" -> "JSON report".
- Tests updated:
  - `src/report/tests.rs`: `serde_yaml` -> `serde_json`. Body assertions: `body.contains("schema-version:")` -> `body.contains("\"schema-version\":")`. `body.contains("spend-usd: null")` -> `body.contains("\"spend-usd\": null")`. The `kebab-case` test still passes; JSON keys are still kebab-case. Rename the test functions: `write_yaml_round_trips` -> `write_json_round_trips`, `yaml_uses_kebab_case_keys_and_emits_jsonl_paths` -> `json_uses_kebab_case_keys_and_emits_jsonl_paths`.
  - `src/render/tests.rs`: same `serde_yaml` -> `serde_json` swap. The `build_context_block` tests parse with `serde_json::Value`. The seeded YAML strings become JSON strings.
  - `src/persona/tests.rs`: switch `serde_yaml::to_string` to `serde_json::to_string`. Update assertions for JSON syntax.
  - `src/tests.rs`: e2e test writes/reads JSON; the temp output filename changes from `.yml` to `.json`.
- `Cargo.toml`: drop `serde_yaml` dependency. Update `description`.
- `.gitignore`: add a third explicit pattern `*claude-report*.json` alongside the existing `*claude-report*.yml` and `*claude-report*.md`. Keep the per-extension structure â€” do not generalize to `*claude-report.*`, which would silently match unrelated files.

Verify: `otto ci` green; `cr collect` then `jq '.totals.spend-usd' <output>` returns the expected number; `cr render` reads the JSON and produces markdown.

Commit: `refactor: emit report as JSON instead of YAML`.

#### Phase 3: prompt synthesis rules
**Model:** opus

Direct prompt rewrite â€” model choice matters because the rule must be unambiguous to a downstream Opus invocation.

- `templates/report.pmt`:
  - Replace the line at ~103 (`Cite session titles and per-session spend where useful...`) with this paragraph:

    > **Synthesize, don't enumerate.** A reader does not want to see every session title with its dollar amount tagged on. Within each tier, pick at most 3-5 representative session titles, chosen for what they convey, not for their cost. Inline dollar amounts in prose are sparing: at most one or two per section, called out only when the number itself is the point (a single session that took 30% of the period spend, an outlier, etc.). The Cost Summary table at the top is the canonical numeric content of this document; do not repeat its numbers in prose. If a tier has 19 sessions, name the theme and cite a few exemplars; do not list all 19. Two sanctioned places remain for per-session dollar amounts: the per-repo summary line (one summary per repo, format `<repo-slug> (N sessions, M tokens, $S spend): ...`) and the Outlier Sessions table.

    The replacement explicitly distinguishes between *per-repo line metrics* (one summary per repo with `(N sessions, M tokens, $S spend)` â€” already in the section, kept as-is) and *per-session enumeration in prose* (the banned pattern).
  - Update the "Numbers that ARE allowed" paragraph (~line 30). Current text:

    > Numbers that ARE allowed are only the ones drawn directly from the context block: total spend, per-model spend, session counts, token counts, repo counts, dates, and exact session titles. That is the entire numeric vocabulary of this document.

    Replace with:

    > Numbers that ARE allowed are only the ones drawn directly from the context block: total spend, per-model spend, session counts, token counts, repo counts, and dates. Per-session dollar amounts are restricted to two sanctioned places: the per-repo summary line in "What This Funded" and the Outlier Sessions table in "Usage Profile". Anywhere else, name the work qualitatively and cite session titles without their dollar amount. That is the entire numeric vocabulary of this document.
  - Add to "Strict rules" (~line 135): "Do not list sessions in bulk with their per-session dollar amounts tagged in parentheses; that pattern is what the synthesis rule above forbids. The per-repo summary line and the Outlier Sessions table are the two sanctioned places for per-session dollar amounts; nowhere else."
  - Update line 144 from `# Context block follows` to `# Context block (JSON) follows`. The prompt currently does not say "yaml" anywhere; only the implicit format hint in the example structures, which become JSON examples below.
  - The "What This Funded" section description at line 90-103 currently says `Prefer bulleted lists with metrics in parentheses over running prose. Format example (use this shape): - <repo-slug> (N sessions, M tokens, $S spend): one-line factual summary citing 1-3 session titles.` This shape is correct (one repo, one summary, three numbers, 1-3 titles cited). The problem was the inner per-session enumeration, not the per-repo line. The synthesis rule reinforces: per-repo line stays; bulk inner enumeration goes.

Verify by running `cr render` and reading the result. Acceptance criterion: no section in the markdown has more than 5 session titles; no section has more than 2 inline dollar amounts in prose; no `(title) ($spend), (title) ($spend), ...` lists exist.

Commit: `docs(prompt): force synthesis over per-session dollar enumeration`.

## Alternatives Considered

### Alternative 1: Keep YAML, document that `yq` is the consumer tool

- **Description:** Leave the YAML serializer in place. Document `yq` in `--help` instead of `jq`. Update the README to spell out the `yq` queries.
- **Pros:** Zero code change. YAML is more human-readable in a terminal, useful when eyeballing the report directly.
- **Cons:** Forces consumers to install `yq`, which is significantly less common than `jq`. The user explicitly chose JSON for portability and ecosystem reach.
- **Why not chosen:** User direction.

### Alternative 2: Emit both YAML and JSON

- **Description:** `cr collect` writes both `.yml` and `.json` outputs (or a `--format` flag selects).
- **Pros:** No clean break; existing YAML consumers keep working.
- **Cons:** Two files per run, twice the path management. A `--format` flag adds surface area and tests. There is no real consumer of the YAML output today other than `cr render` itself, so dual output protects no one.
- **Why not chosen:** Speculative flexibility for an audience of one. Drop it; if someone needs YAML later, add a flag at that point.

### Alternative 3: Make `jq` optional, only fail if a `--validate` flag is passed

- **Description:** `cr collect` does not enforce `jq` presence. A future `cr validate` subcommand would shell out to `jq` and assert query roundtrips.
- **Pros:** No new hard dependency. cr produces JSON regardless of whether `jq` is on PATH.
- **Cons:** Defers the enforcement point. The user wants the `cr collect` output to be queryable as a top-level value proposition; if `jq` is missing, the produced file is essentially write-only.
- **Why not chosen:** User direction. The scope is enforce on `cr collect`; that is the producer-side check that catches the gap early.

### Alternative 4: Soften jq enforcement to a stderr warning instead of failing `cr collect`

- **Description:** When `jq` is missing, print a loud warning to stderr (`cr collect: warning: jq not found on PATH; the JSON report will be queryable only with custom code or yq/python/etc.`) but proceed with report generation.
- **Pros:** Does not break automation that runs `cr collect && cr render` headlessly without ever invoking `jq`. Architect raised this concern explicitly in design review: `cr` does not invoke `jq` itself, so failing on its absence is producer-side enforcement of a consumer-side tool, which is unusual.
- **Cons:** Loses the early-detection property. A user who installs cr fresh, never sees the warning (because they have stderr suppressed in their workflow), runs `cr collect`, gets a JSON file, then tries `jq '...' <file>` and discovers `jq: command not found` at the worst possible time.
- **Why not chosen:** Conscious user direction. The audience for v0.1.2 is one-author + small-circle internal use, not headless automation pipelines. The fail-fast posture matches that audience's expected workflow (interactive reports queried at the shell). If a CI/CD use case emerges later, revisiting this is straightforward â€” the check is one block in `main.rs`. Architect concern noted and explicitly conceded.

### Alternative 5: Add the synthesis rule by tightening `report.pmt`'s "Hard prohibition" section instead of adding a new section

- **Description:** Extend the existing "Hard prohibition: no speculative quantification" with bullets that ban per-session enumeration.
- **Pros:** One section to maintain.
- **Cons:** Conflates two distinct rules. Speculative quantification (banning $ vs engineer-cost comparisons) is about credibility; synthesis (banning per-session enumeration) is about readability. Putting both under one heading dilutes both.
- **Why not chosen:** Keep them as separate, named rules so model and reader understand they are different commitments.

## Technical Considerations

### Dependencies

- **Add:** `which` (small crate, used to find `jq` on PATH at startup).
- **Drop:** `serde_yaml`. Nothing in cr will need YAML after Phase 1.
- **Unchanged:** `serde_json` already in tree.

### Performance

**On-disk serialization:** `serde_json::to_string_pretty` and `serde_yaml::to_string` are both linear in record count. For 429-session reports the wall-clock cost is dominated by file I/O. JSON pretty-printed is structurally noisier than YAML (quotes around keys, braces, commas) â€” expect the on-disk file to grow modestly (rough estimate: ~328kB YAML â†’ ~400kB JSON pretty-printed for the April 2026 report). Acceptable.

**LLM context-block token cost:** the wrapper sent to Opus uses **compact** JSON (`serde_json::to_string`, no whitespace), not pretty. This addresses the single largest cost concern raised in the Architect review: had the prompt embedded pretty JSON, every `cr render` invocation would have paid for an extra ~18k input tokens vs. the YAML baseline (~$0.09 per render at Opus input pricing). Compact JSON closes the gap to near zero â€” JSON's structural noise is mostly the indentation, not the punctuation. The split between compact-in-context and pretty-on-disk costs nothing structurally: the same `Report` value is serialized twice for two consumers with different needs.

### Security

No new attack surface. `which::which("jq")` reads `$PATH` and stats files. The `jq` invocation never happens inside cr; cr only checks whether the binary exists, then bails or proceeds. Users invoke `jq` themselves on cr's output.

### Testing Strategy

- `report/tests.rs` and `render/tests.rs` round-trip JSON: write a `Report`, read it back, assert structural equality.
- `report/tests.rs::yaml_uses_kebab_case_keys_and_emits_jsonl_paths` keeps its name? Rename to `json_uses_kebab_case_keys_...` to avoid drift. Update assertions to `body.contains("\"schema-version\":")` etc.
- `persona/tests.rs` round-trips persona JSON and asserts the omitted-when-None behavior.
- `render/tests.rs::build_context_block_*` parses the context block as `serde_json::Value` and asserts structure.

E2E:
- `tests.rs::end_to_end_collect_writes_json` (renamed) writes a temp projects dir, runs `crate::run`, parses `claude-report.json` with `serde_json`, asserts session count, spend, etc.
- `tests.rs::end_to_end_title_preserved_across_runs` keeps existing semantics, file extension changes.

Manual verification:
- Run `cr collect` against the snapshot at `/tmp/cr-baseline-snap`; assert the JSON file is jq-queryable: `jq '.totals.sessions' <out>` returns the same 445 number reported pre-migration.
- Run `cr render`; read the markdown; verify no section has more than 5 session titles or more than 2 inline dollar amounts.
- `mv ~/.cargo/bin/jq /tmp/.jq.bak && cr collect` â†’ expect the install hint and exit non-zero. Restore the binary afterward.

### Rollout Plan

Direct push to `main`. No feature flag. v0.1.0 (the initial release with stale workspace files) and v0.1.1 (the first clean release) keep emitting YAML. v0.1.2 emits JSON. Anyone holding an old YAML report gets a clear failure on `cr render` (`failed to parse report at ...`); their fix is `cr collect` again.

**Title preservation across the upgrade.** `report::load_existing_titles` reads the existing report file at the output path to carry titles forward across `cr collect` runs. After the format switch, `load_existing_titles` reads JSON from `./claude-report.json`. A user upgrading from v0.1.1 with an existing `./claude-report.yml` on disk will not have their titles preserved on the first v0.1.2 run, because the loader looks at the `.json` path which doesn't exist yet. After the first `cr collect` on v0.1.2, titles persist normally. This is a one-time loss and acceptable; documenting it in the commit message.

### Architect Review Outcomes (2026-04-29)

Architect raised four findings in design review. Resolution:

- **Title-loss on upgrade:** Conceded. v0.1.2 is a clean break with no YAML fallback in `load_existing_titles`. CLAUDE.md forbids backward-compat shims when a clean change works; the cost of one re-titling pass via Haiku at ~$0.01/session is acceptable.
- **JSON token cost in Opus context:** Actioned. Compact JSON for the LLM context block, pretty JSON for on-disk. Documented in the Performance section.
- **Legacy `-i` extension detection:** Actioned. `render::run` checks for `.yml`/`.yaml` input paths and bails with a guided error before invoking `serde_json::from_str`.
- **`.gitignore` explicitness:** Actioned. Add a third explicit pattern `*claude-report*.json` rather than generalizing the existing patterns.
- **`jq` enforcement is producer-side overreach:** Conceded with rationale. Captured as Alternative 4. Author's call: fail-fast is the right posture for the intended audience (interactive shell use); the check is one block in `main.rs` and easy to soften later if a headless use case appears.
