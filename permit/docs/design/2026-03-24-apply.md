# Design Document: apply subcommand

**Author:** Scott Idler
**Date:** 2026-03-24
**Status:** Implemented
**Review Passes Completed:** 3/3

## Summary

Add an `apply` subcommand to `claude-permit` that executes audit recommendations by modifying `settings.json` and `settings.local.json`. Supports applying all recommendations or filtering by recommendation type (promote, remove, narrow, deny).

## Problem Statement

### Background

`claude-permit audit` produces actionable recommendations (promote, remove, narrow, deny) but the user must manually edit JSON files to act on them. With 157 promote and 44 remove recommendations in the current setup, manual editing is impractical.

### Problem

There is no way to act on audit recommendations without hand-editing settings files.

### Goals

- Apply audit recommendations to settings files automatically
- Support filtering by recommendation type (e.g., only apply "promote" or only "remove")
- Always show what will change before writing (dry-run by default)
- Create backups before modifying files
- Preserve all non-permission fields in settings files (env, model, hooks, etc.)

### Non-Goals

- Interactive per-rule approval (too slow for 157+ rules - use filters instead)
- Implementing the "narrow" recommendation (requires human judgment to decide how to split a broad rule)
- Undo/rollback (backups are sufficient)

## Proposed Solution

### Overview

The `apply` subcommand runs the same audit logic, then partitions recommendations into operations on the two settings files. It modifies only the `permissions.allow` and `permissions.deny` arrays, preserving everything else via serde_json `Value` manipulation (not struct deserialization).

### CLI Design

```
claude-permit apply [OPTIONS]
  --promote          Apply "promote" recommendations (move safe local rules to global)
  --remove           Apply "remove" recommendations (delete dangerous local rules)
  --deny             Apply "deny" recommendations (add permanently-denied patterns to deny list)
  --all              Apply all actionable recommendations (promote + remove + deny)
  --settings <PATH>        Override settings.json path
  --settings-local <PATH>  Override settings.local.json path
  --dry-run          Show what would change without writing (DEFAULT)
  --yes              Actually write changes (required to make modifications)
  --no-backup        Skip creating .bak files before writing
```

Key design decisions:
- **Dry-run is the default.** User must pass `--yes` to actually write. This prevents accidental modifications.
- **At least one filter flag is required** (`--promote`, `--remove`, `--deny`, or `--all`). Running bare `apply` with no filter shows usage help.
- **`--narrow` is deliberately omitted.** The tool cannot know how to split a broad pattern - it prints a message suggesting the user do this manually.

### Operations by Recommendation Type

**Promote (local -> global):**
1. If the rule already exists in `settings.json` `permissions.allow`, just remove it from local (dedup)
2. Otherwise, add the rule to `settings.json` `permissions.allow` and remove from `settings.local.json` `permissions.allow`

**Remove (delete from local):**
1. Remove the rule from `settings.local.json` `permissions.allow`

**Deny (remove from allow lists):**
1. Remove the rule from `permissions.allow` in whichever file contains it
2. Do NOT add to `permissions.deny` automatically - the built-in deny list in settings.json already covers the standard patterns (git tag -d, etc.), and the deny format may differ from the allow format (e.g., `Bash(rm -rf *)` vs `Bash(rm -rf:*)`)
3. Print a note suggesting the user add an explicit deny entry if desired

### File Modification Strategy

Use `serde_json::Value` to load the entire JSON, modify only the `permissions.allow` and `permissions.deny` arrays, then write back with `serde_json::to_string_pretty`. This preserves all other fields (env, model, hooks, statusLine, plugins, etc.).

```rust
// Pseudocode
let mut global: Value = serde_json::from_str(&read(settings_path))?;
let mut local: Value = serde_json::from_str(&read(settings_local_path))?;

// Modify permissions arrays
apply_operations(&mut global, &mut local, &operations);

// Write back with pretty printing
write(settings_path, serde_json::to_string_pretty(&global)?)?;
write(settings_local_path, serde_json::to_string_pretty(&local)?)?;
```

### Output Format

Dry-run output (default):
```
Would apply 157 promote, 44 remove, 0 deny operations:

PROMOTE (local -> global): 157 rules
  + Bash(ls:*)
  + Bash(tree:*)
  + WebSearch
  ... (154 more)

REMOVE (from local): 44 rules
  - Bash(sudo rm:*)
  - Bash(sudo apt update:*)
  ... (42 more)

Skipped: 1 narrow (requires manual review)

Pass --yes to apply these changes.
```

Applied output (with --yes):
```
Backup: ~/.claude/settings.json.bak
Backup: ~/.claude/settings.local.json.bak

Applied 157 promote, 44 remove, 0 deny operations.
  settings.json: 157 rules added to allow
  settings.local.json: 201 rules removed from allow
```

### Implementation Plan

**Single phase** - this is a focused addition:
- Add `Apply` variant to `Command` enum in `cli.rs`
- Create `src/cmd/apply.rs` with the apply logic
- Add settings file writer to `src/settings/parser.rs` (or a new `writer.rs`)
- Wire up in `main.rs`
- Tests for each operation type and edge cases

## Alternatives Considered

### Alternative 1: Interactive per-rule approval
- **Description:** Prompt y/n for each rule
- **Pros:** Maximum control
- **Cons:** Impractical for 200+ rules, doesn't work in non-interactive contexts
- **Why not chosen:** Filter flags provide sufficient granularity without tedium

### Alternative 2: Generate a patch/diff file
- **Description:** Output a JSON patch that the user applies manually
- **Pros:** Maximum safety, user reviews exact changes
- **Cons:** Adds friction, JSON patches are hard to read
- **Why not chosen:** Dry-run default achieves the same review step with less friction

### Alternative 3: Edit in place with jq
- **Description:** Shell out to jq for JSON manipulation
- **Pros:** Proven tool
- **Cons:** External dependency, harder to test, jq syntax is fragile for this
- **Why not chosen:** serde_json::Value is simpler and already a dependency

## Technical Considerations

### Dependencies

No new dependencies - `serde_json::Value` manipulation is already available.

### Security

- Backup files created before any write
- Dry-run is the default - `--yes` required for writes
- Never modifies deny list entries that already exist
- Write directly after creating backup - backup is the safety net

### Testing Strategy

- Unit tests for each operation type (promote, remove, deny) on in-memory JSON Values
- Integration test: load test fixtures, apply operations, verify resulting JSON
- Edge case: rule exists in both global and local
- Edge case: empty permissions arrays
- Edge case: settings file with no permissions block at all
- Edge case: settings.local.json does not exist (promote/remove are no-ops)
- Edge case: rule exists in both global and local allow lists (dedup on promote)
- Verify output JSON uses 2-space indent with trailing newline

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Pretty-printing changes JSON formatting/key order | Medium | Low | serde_json preserves insertion order; formatting may change but is correct |
| Backup file overwrites previous backup | Low | Medium | Include timestamp in backup filename: `.bak.YYYYMMDD-HHMMSS` |
| Race condition if Claude Code writes settings during apply | Low | Medium | Warn user to not run during active session; backup provides recovery |

## Open Questions

All resolved during review:
- Filters are always explicit - `--all` is never default (prevents accidents)
- Duplicate rules are handled by promote dedup logic (just remove from local if already in global)
